# Full SEO Audit Report — Tanzania Sensational

**Date:** 2026-05-05  
**Auditor:** SEO Lead  
**Site:** tanzaniasensational.com  
**Stack:** Laravel 11 (API) + React 18 SPA (CSR) + Vite 7

---

## Executive Summary

This site has strong foundations — server-rendered meta tags, dynamic sitemap, JSON-LD schema injection, and a well-structured content model. However, **the single biggest issue is that this is a pure Client-Side Rendered (CSR) SPA**, which creates significant indexability risk. Google can render JS, but it does so on a second pass and may not execute all JavaScript, especially for deeply nested content or pages with heavy API dependencies. Combined with thin blog content, missing meta fields on key models, and a bare-bones sitemap, the site is leaving substantial organic visibility on the table.

**Estimated organic traffic impact if all Critical/High issues are fixed:** 5-10x increase within 6 months (from current ~10/mo to 50-100/mo), assuming content investment.

## 1. TECHNICAL SEO

### 1.1 CRITICAL: CSR-Only Rendering (Indexability Risk)

| Issue | Severity | Details |
|-------|----------|---------|
| Pure CSR SPA | **CRITICAL** | The site uses `ReactDOM.createRoot()` on a minimal Blade shell (`resources/views/app.blade.php`). All content is loaded via API calls after JS execution. Google's rendering queue may not execute all JS, especially for pages requiring API data. |

**Evidence:**
- [`resources/js/main.jsx`](resources/js/main.jsx) — `ReactDOM.createRoot(document.getElementById('root')).render(...)` — pure CSR mount
- [`resources/views/app.blade.php`](resources/views/app.blade.php) — only `<div id="root"></div>` in body
- All page components (e.g., [`Lemosho.jsx`](resources/js/pages/trekking/kilimanjaro/Lemosho.jsx:25), [`Machame.jsx`](resources/js/pages/trekking/kilimanjaro/Machame.jsx:25)) fetch data via `useEffect` → API calls

**Impact:** Google may index only the default meta tags and an empty shell for pages where JS fails to execute or API calls time out. This is the #1 reason this site has only ~10 monthly organic visits.

**Recommendation:** Implement SSR (via Laravel + Inertia.js, or a lightweight prerendering service like Prerender.io). At minimum, add server-side rendered fallback content for critical pages.

### 1.2 HIGH: Sitemap Lacks Priority, Changefreq, and Lastmod

| Issue | Severity | Details |
|-------|----------|---------|
| Incomplete sitemap | **HIGH** | The sitemap at `/sitemap.xml` only includes `<loc>` elements. No `<priority>`, `<changefreq>`, or `<lastmod>` tags. |

**Evidence:** [`resources/views/sitemap.blade.php`](resources/views/sitemap.blade.php:4-6) — only `<loc>{{ $u['loc'] }}</loc>` per URL

**Impact:** Google uses `<lastmod>` to determine recrawl frequency. Without it, fresh content (new blog posts, updated routes) may not be crawled promptly.

**Recommendation:** Add `<lastmod>`, `<changefreq>`, and `<priority>` to each URL entry. Use model `updated_at` timestamps for dynamic content.

### 1.3 HIGH: No Hreflang Tags Despite Multi-Market Targeting

| Issue | Severity | Details |
|-------|----------|---------|
| Missing hreflang | **HIGH** | The site targets 12+ countries (UK, Germany, France, Netherlands, Italy, Spain, US, Canada, Australia, Japan, China, Singapore, UAE) but has zero hreflang tags. |

**Evidence:** [`resources/views/app.blade.php`](resources/views/app.blade.php:1-38) — no `<link rel="alternate" hreflang="..." href="...">` tags

**Impact:** Google may serve the English page to German searchers, or worse, treat multi-language content as duplicate. This is critical for a site targeting 12+ countries.

**Recommendation:** Implement hreflang for target languages (en, de, fr, it, nl, es, ja, zh). Add `x-default` for the English version.

### 1.4 MEDIUM: Robots.txt Allows Everything

| Issue | Severity | Details |
|-------|----------|---------|
| Permissive robots.txt | **MEDIUM** | `robots.txt` allows all crawlers to all paths. |

**Evidence:** [`robots.txt`](robots.txt:1-2):
```
User-agent: *
Disallow:
```

**Impact:** Admin routes (`/ops-7f3d/*`) and API endpoints are crawlable. While `.htaccess` blocks some paths, `robots.txt` should explicitly disallow admin and API paths to prevent crawl waste.

**Recommendation:**
```
User-agent: *
Disallow: /ops-7f3d/
Disallow: /api/
Disallow: /storage/
```

### 1.7 LOW: No HTTP → HTTPS Redirect Enforced in .htaccess

| Issue | Severity | Details |
|-------|----------|---------|
| Missing HTTPS redirect | **LOW** | The `.htaccess` does not include a rule to redirect HTTP to HTTPS. |

**Evidence:** [`..htaccess`](.htaccess:1-44) — no `RewriteCond %{HTTPS} off` rule

**Impact:** If users or crawlers access the site via HTTP, they won't be redirected. This is low severity if the server-level config already handles this (common with cPanel/Forge).

**Recommendation:** Verify server-level HTTPS enforcement. If not present, add:
```
RewriteCond %{HTTPS} off
RewriteRule ^(.*)$ https://%{HTTP_HOST}/$1 [R=301,L]
```

**Impact:** 4 font families with 20+ weights = significant render-blocking. This will negatively impact LCP.

**Recommendation:** Consider reducing to 2 font families. Add `preconnect` for Google Fonts origin (already present). Use `font-display: swap` (already present via `&display=swap`).
